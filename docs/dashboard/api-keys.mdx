<CodeGroup>
```typescript API Key Generation
import { generateApiKey } from '@/lib/api/auth'
import { useMutation } from 'convex/react'
import { api } from '@/convex/_generated/api'

const createKey = useMutation(api.apiKeys.create)

const handleGenerateKey = async () => {
  // Generate cryptographically secure API key
  const { apiKey, hashedKey, keyPrefix } = generateApiKey()

  // Save hashed version to database
  await createKey({
    userId: user._id,
    hashedKey,        // SHA-256 hash (never store plain key)
    keyPrefix,        // First 8 chars for display (e.g., "gs_live_")
    tier: newKeyTier,
    rateLimit: rateLimits[newKeyTier],
    name: newKeyName,
  })

  setGeneratedKey(apiKey) // Show to user ONCE
}
```

```typescript Rate Limits by Tier

}

const dailyLimits = {
  startup: 1000,    // 1K requests per day

  enterprise: null, // Unlimited
}
```

- Environment variables (`.env` file, never committed to git)

</Warning>

```typescript API Key Data Model
interface ApiKey {
  _id: string
  userId: string
  name?: string
  keyPrefix: string        // First 8 chars (e.g., "gs_live_")
  hashedKey: string        // SHA-256 hash (for verification)
  tier: 'startup' | 'growth' | 'enterprise'
  rateLimit: number        // Requests per minute
  isActive: boolean        // false if revoked
  createdAt: number        // Unix timestamp
  lastUsedAt?: number      // Unix timestamp of last API call
}
```

### Revoke Confirmation

**Trigger**: Click "Revoke" button on active API key

**Flow**:

<Steps>
  <Step title="Browser Confirm Dialog">
    Native browser confirm: "Are you sure you want to revoke this API key? This cannot be undone."
  </Step>
  <Step title="User Confirms">
    If yes, proceed to revocation
  </Step>
  <Step title="Mutation Call">
    `revokeKey({ apiKeyId, userId })` mutation executed
  </Step>
  <Step title="Database Update">
    API key marked as `isActive: false` in Convex
  </Step>
  <Step title="Success Toast">
    "API key revoked successfully" notification
  </Step>
  <Step title="Card Updates">
    Key card shows "Revoked" badge, Revoke button disappears
  </Step>
</Steps>

<Info>
**Immediate Effect**: Revoking an API key immediately invalidates it. Any subsequent API requests using that key will receive a `401 Unauthorized` error.
</Info>

## Rate Limiting Details

### Tier Comparison

| Tier | Cost/Month | Rate Limit | Daily Limit | Best For |
|------|------------|------------|-------------|----------|
| **Startup** | $29 | 10 req/min | 1K req/day | Small apps, prototypes |
| **Growth** | $99 | 60 req/min | 20K req/day | Growing businesses |
| **Enterprise** | $499 | 300 req/min | Unlimited | High-volume integrations |

### Rate Limit Headers

All API responses include rate limit headers:

```bash
HTTP/1.1 200 OK
X-RateLimit-Limit: 10           # Requests allowed per minute
X-RateLimit-Remaining: 7        # Requests remaining in window
X-RateLimit-Reset: 1735689600   # Unix timestamp when limit resets
```

### Exceeding Rate Limits

**Response when limit exceeded**:

```json
{
  "error": "Rate limit exceeded",
  "message": "You have exceeded your rate limit of 10 requests per minute",
  "retryAfter": 30,  // Seconds until reset
  "tier": "startup"
}
```

**HTTP Status**: `429 Too Many Requests`

<Warning>
**Handling 429 Errors**: Implement exponential backoff in your application:
1. Catch 429 status code
2. Read `retryAfter` value from response
3. Wait specified seconds before retrying
4. Consider caching Ghost Score results to reduce API calls
</Warning>

## Common Questions

<AccordionGroup>
  <Accordion title="Can I regenerate a lost API key?">
    **No** - API keys are shown only once during generation for security reasons.

    **Solution**:
    1. Revoke the lost key immediately (prevent unauthorized use)
    2. Generate a new API key
    3. Update your applications with new key
    4. Store new key securely this time

    **Best Practice**: Use secret management tools (AWS Secrets Manager, Vault) to avoid losing keys.
  </Accordion>

  <Accordion title="How many API keys can I create?">
    **Limit**: 10 active API keys per account

    **Reason**: Prevents abuse and encourages proper key rotation

    **Workaround**: Revoke unused keys to free up slots

    **Enterprise Exception**: Contact support@ghostspeak.io for higher limits
  </Accordion>

  <Accordion title="Are API keys tied to specific agents?">
    **No** - API keys are account-level credentials:

    - One API key can verify any agent on the network
    - Rate limits apply across all agents checked
    - Billing is per API key tier, not per agent

    **Use Case**: Generate one key per application/environment, not per agent.
  </Accordion>

  <Accordion title="What's the difference between B2C and B2B pricing?">
    **B2C (Consumer)**:
    - Pay per verification via dashboard
    - USDC or GHOST token payment
    - No API access
    - Designed for one-off verifications

    **B2B (API Keys)**:
    - Monthly subscription ($29-$499)
    - Programmatic access via REST API
    - Rate limits based on tier
    - Designed for integrations

    **Recommendation**: Use B2B if you need >10 verifications/month or automated checks.
  </Accordion>

  <Accordion title="Can I test the API without paying?">
    **Yes!** On devnet:
    - Generate API keys for free
    - No rate limits enforced (devnet only)
    - Test all endpoints without cost

    **Mainnet** (Q1 2026):
    - All API keys require paid subscription
    - 7-day free trial available for Startup tier
    - No free tier on mainnet
  </Accordion>
</AccordionGroup>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Never commit API keys to git">
    **Bad**:
    ```javascript
    // ❌ NEVER DO THIS
    const API_KEY = 'gs_live_abc123...'
    ```

    **Good**:
    ```javascript
    // ✅ Use environment variables
    const API_KEY = process.env.GHOSTSPEAK_API_KEY
    ```

    **Tools**:
    - Add `.env` to `.gitignore`
    - Use `dotenv` package for local development
    - Use platform secrets (Vercel, Railway) for production
  </Accordion>

  <Accordion title="Rotate keys regularly">
    **Frequency**: Every 90 days for production keys

    **Process**:
    1. Generate new API key
    2. Update production environment variables
    3. Deploy updated configuration
    4. Revoke old key after 24-hour grace period
    5. Monitor for errors from old key usage
  </Accordion>

  <Accordion title="Use different keys per environment">
    **Recommended Setup**:
    - **Development**: One key for local testing
    - **Staging**: Separate key for staging environment
    - **Production**: Dedicated key for live app

    **Benefit**: If staging key compromised, production unaffected
  </Accordion>

  <Accordion title="Monitor API key usage">
    Visit `/dashboard/api-usage` to track:
    - Request count per key
    - Error rates
    - Last used timestamp
    - Geographic usage patterns

    **Alert on**:
    - Unusual spike in requests (possible abuse)
    - Requests from unexpected IPs
    - High error rates (misconfigured integration)
  </Accordion>
</AccordionGroup>
